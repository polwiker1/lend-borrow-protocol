Que modela

Lending Tango Parana muestra como un activo real puede documentarse, tokenizarse y usarse como base de credito. No es un producto financiero final: es una maqueta auditable para que alumnos, cooperativas, lenders, auditores y perfiles financieros entiendan la logica.

Financiamiento productivo

El tomador deja activo agro / agTICKET como respaldo y recibe liquidez estable.

Exposicion a trigo tokenizado

El tomador deja garantia estable y recibe wTK para quedar expuesto al precio del trigo.

Logistica portuaria

Financia transporte, puerto, seguros, combustible o fletes. Puede pedir aUSD o wTK.

Actores

  • Administrador / multisig: crea LTP, emite tickets, fondea liquidez y aprueba SLT.
  • Borrower: recibe el LTP, deposita colateral, toma deuda y firma la ejecucion final.
  • Wallet 3: recibe liquidez por una cesion de margen SLT, sin controlar el LTP original.
  • Auditor: revisa hashes, estados, contratos, direcciones y eventos on-chain.
  • Liquidador: actua cuando una posicion cae debajo del umbral de seguridad.

Flujo LTP

  1. El administrador crea un ticket primario LTP con datos de actor, activo, monto, plazo y documento.
  2. El LTP se emite a la wallet del borrower.
  3. El borrower bloquea el LTP en el vault.
  4. El vault custodia el ERC1155 y emite 1 agTICKET como recibo ERC20.
  5. El borrower deposita agTICKET como colateral en LendingProtocol.
  6. El borrower pide el token prestado: aUSD, wTK, sTK, gTK u otro mercado configurado.

En logistica portuaria, el LTP no tiene que representar grano fisico directo. Puede documentar una operacion auditada de transporte, puerto, acopio, seguros, combustible, mantenimiento o fletes. Si pide aUSD, busca liquidez estable. Si pide wTK, asume deuda vinculada al precio del trigo tokenizado.

Flujo SLT

El SLT no crea credito sin respaldo. Usa margen libre de un LTP ya auditado.

  1. Ejemplo: LTP vale 100, permite 65, el borrower ya uso 50 y quedan 15 libres.
  2. El borrower solicita un SLT para ceder parte de ese margen a una wallet 3.
  3. El administrador aprueba el SLT luego de revisar documento y riesgo.
  4. El borrower ejecuta el desembolso con su firma.
  5. La wallet 3 recibe el token prestado, pero la deuda sigue atada al borrower/LTP.

Riesgo visible

  • Factor colateral: define cuanto se puede pedir contra una garantia.
  • Umbral de liquidacion: define cuando una posicion puede ser liquidada.
  • Oracle: alimenta el precio, pero no reemplaza auditoria ni gobierno de riesgo.
  • SLT: aumenta deuda total del LTP, por eso debe verse separado y sumado.
  • Bad debt: si el colateral cae demasiado rapido, el protocolo debe mostrar la perdida.

Como probar

  1. Conectar wallet administrador y crear/emitar LTP.
  2. Conectar wallet borrower y bloquear LTP en vault.
  3. Depositar agTICKET como colateral.
  4. Fondear liquidez del token prestado desde el administrador.
  5. Pedir el token prestado desde el borrower.
  6. Para SLT: solicitar con borrower, aprobar con administrador y ejecutar con borrower.

Cada usuario prueba con su propia wallet. El panel no guarda claves privadas y solo firma transacciones desde MetaMask o una wallet compatible. Para ver saldos, importa aUSD, wTK, sTK, gTK y agTICKET desde la tarjeta "Contratos para importar en wallet".

LTP vs SLT

LTP es el ticket primario de la operacion original. SLT es un ticket de mercado secundario sobre tokens WRA que ya arrastran riesgo financiero. Por eso aplica un limite didactico del 65%.
